# Протокол дисплея Zalman Alpha 2 (реверс Zalman OZ)

Восстановлено из `LcdComm.dll` (нативный x86-64) + USB-дампов (USBPcap) рабочего
Windows-приложения. Всё подтверждено на живом устройстве.

## Общее

- Приложение Zalman OZ — **.NET/WPF**, не Lua (найденные `*.lua` — это встроенный
  плеер VLC для видео-тем, к протоколу отношения не имеют).
- Устройство: USB **`VID 0483 : PID 5740`** («USB Display», STM CDC). Второе
  устройство `0145:2001` — контроллер помпы/вентиляторов/RGB (HID), не дисплей.
- Экран **320×320**.
- Транспорт: **bulk OUT, endpoint `0x02`**. В драйвере используется **libusb**
  напрямую (в обход `cdc_acm`) — так проще контролировать целостность передач.
- Ответы (identify) приходят на bulk IN `0x82`.

## Пробуждение (verify) — обязательно после холодного старта

Пока хост не «поздоровался», экран тёмный. Последовательность (bulk OUT):

1. `01` + `"HWCX-TECH-VRFY0"`  (16 байт)
2. `01` + `"HWCX-TECH-VRFY1"`  (16 байт)
3. 16-байтовый challenge: `[01, r1..r15]`, где r — случайные, затем поля
   переопределяются: `b[8]=(b[5]+b[6])^b[7]`, `b[4]=(b[2]+b[3])^b[1]`,
   `b[9]=~b[10]`, контрольная сумма `b[11]=(-sum(b[1..10]))&0xFF`.

Устройство отвечает 17 байтами `"HWCX-TECH-320x320"` (модель). Нужно также
поднять DTR/RTS (при работе через tty) — при libusb не требуется.

## Пиксель и слои

- Пиксель — 32 бита **`(A<<24)|(R<<16)|(G<<8)|B`** (BGRA), **A — альфа**.
  A=0xFF непрозрачный, A=0 прозрачный (пропускает нижний слой).
- Два слоя: **фон** (кадры `0x05`) и **оверлей** (`0x07`), альфа-композитятся.

## Команды

### 0x05 — кадр фона (JPEG) ★ плавное видео

```
[05,00,00,00, jpeglen(u32 LE), 0×8]   (16 байт заголовок, отдельная передача)
<JPEG-данные>                          (ffd8…ffd9), добитые нулями до кратности 64
```
JPEG сжат (~10 КБ против ~400 КБ сырого) → по USB full-speed идёт ~30–60 к/с,
поэтому видео/GIF **плавные**. Кадры стримятся подряд; фон-слой держится.

### 0x07 — оверлей (RLE + альфа)

```
[07, sub, 0,0, clen(u32 LE), x(u16) y(u16) w(u16) h(u16)]   (16 байт)
<RLE-тело> + терминатор 00 00 00 00
```
- `sub` 0x03 (два sub-буфера складываются при показе — писать надо в ОДИН).
- `x=y=0, w=h=320` (полный кадр; **частичные регионы НЕ использовать — вешают
  парсер прошивки, лечится только power cycle**).
- RLE по 32-битным пикселям:
  - RUN: `0x02000000 | count`, затем 1 пиксель (повторить count раз);
  - LITERAL: `0x01000000 | count`, затем count пикселей;
  - конец — токен `0x00000000`.
- Прозрачный фон (A=0) + непрозрачный текст (A=0xFF) → строка ложится поверх
  фона. Оверлей держится и композитится над кадрами `0x05`.

### 0x08 — яркость / поворот

```
[08, rotate, brightness, 0…]   (16 байт)
```
`0xFF` в поле = «не менять». Яркость масштабируется `v*90/100+10`. (В драйвере
поворот делается программно, а не этой командой.)

### 0x02 / 0x06 — заливка темы во flash (не используется драйвером)

`[02, fps, count(u16 BE)]` + count×(`0x05`+JPEG) + `[06,…]`. Заливает тему во
flash; она проигрывается **после перезагрузки** устройства. Триггер запуска из
рантайма не найден, поэтому для живого показа используется стрим `0x05`, а не
flash.

## Формат `.th` (штатные темы)

`FF FD×6 FF` + u32 BE: width, height, fps, frameCount, dataOffset; далее таблица
смещений и склеенные JPEG-кадры 320×320.

## Грабли (проверено болью)

- **Частичный регион** в `0x07` (w<320 или h<320) → залипание парсера → power
  cycle. Драйвер всегда шлёт полный кадр.
- **Таймаут bulk-передачи посреди кадра** → недосыл → рассинхрон → залипание
  endpoint. Использовать большой таймаут, не рвать кадр.
- Запись во flash неточным форматом также вешала устройство — не эксперименти-
  ровать на живом железе.
